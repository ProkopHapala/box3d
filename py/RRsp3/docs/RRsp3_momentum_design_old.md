# RRsp3: Momentum-Corrected Projective Dynamics Design

## 1. Overview
The goal is to implement a rigorous Position-Based Dynamics (PBD) / Extended PBD (XPBD) solver for rigid molecular dynamics in `RRsp3`. Key requirements are **conservation of linear and angular momentum** (physical correctness) and **accelerated convergence** (solver efficiency) using Heavy-Ball momentum, matching the reference implementation in `cpp/common/math/ProjectiveDynamics_d.cpp`.

## 2. Physics & Solver Formulation

### 2.1 XPBD vs Projective Dynamics
The current `RRsp3.cl` implements an **XPBD** (Extended PBD) formulation.
*   **XPBD Update**: 
    $$ \Delta \lambda = \frac{-C(x)}{\tilde{\alpha} + \sum w_i \|\nabla C\|^2} $$
    Where $\tilde{\alpha} = \frac{1}{K dt^2}$ is the compliance.
    Positions are updated via $\Delta x_i = w_i \Delta \lambda \nabla_i C$.
*   **Equivalence**: This is mathematically equivalent to an implicit Euler integration step or a single iteration of a Projective Dynamics solver minimizing:
    $$ E(x) = \frac{1}{2dt^2} (x - x_{pred})^T M (x - x_{pred}) + \frac{1}{2} K C(x)^2 $$
*   **Weights**: `RRsp3.cl` correctly uses $w_i = 1/m_i$ and $w_{rot} \approx (r \times n)^T I^{-1} (r \times n)$. The compliance $\alpha$ accounts for the $M/dt^2$ inertial term.

### 2.2 Momentum Conservation (Physical)
To ensure physical momentum conservation during Jacobi iterations:
1.  **Linear Momentum**: The solver uses a **gather** approach. 
    *   Thread $i$ computes violation for bond $i-j$.
    *   Applies correction $\Delta x_i$ to itself.
    *   Writes recoil correction $\Delta x_j$ to `dpos_neigh` for neighbor $j$.
    *   In `apply_corrections`, atom $i$ sums its own `dpos_node` and all `dpos_neigh` aimed at it (via `bkSlots`).
    *   **Result**: Every impulse $P$ generated by bond $i-j$ is applied as $+P$ to $i$ and $-P$ to $j$. Sum is zero. **Linear momentum is conserved.**
2.  **Angular Momentum**: 
    *   Torques are generated via $\Delta \theta = I^{-1} (r \times P)$.
    *   For conservation, the sum of torques must be zero: $\tau_i + \tau_j = r_i \times P + r_j \times (-P) = (r_i - r_j) \times P$.
    *   Since $P$ acts along the line connecting the *current* port positions (in PBD projection), $P \parallel (p_i - p_j)$.
    *   Ideally $p_i = x_i + r_i$ and $p_j = x_j + r_j$. The vector connecting them is $d = p_j - p_i$.
    *   $P \propto d$. Thus $(p_j - p_i) \times P = 0$.
    *   $(x_j + r_j - x_i - r_i) \times P = 0 \implies (r_j - r_i) \times P = (x_i - x_j) \times P$.
    *   This implies total angular momentum is conserved relative to the system CoM *if* the constraints are satisfied or the projection is exact.
    *   In the iterative solver, we must ensure symmetric application. The current symmetric scatter/gather of linear impulses combined with local torque generation is the standard PBD approach for rigid bodies and approximates conservation well.

### 2.3 Solver Momentum (Acceleration)
To speed up convergence (Jacobi iterations), we apply **Heavy-Ball Momentum** (or Nesterov-like acceleration) to the solver updates. This is distinct from physical velocity.

**Reference (`ProjectiveDynamics_d.cpp`)**:
The reference implementation uses a specific momentum update scheme for the linear solver:
```cpp
// psa = x_k (current iterate)
// psb = x_{k+1}' (prediction from Jacobi step)
// linsolve_yy = d_k (previous step direction)

Vec3d p = psb[j];                    // p = Jacobi(x_k)
p.add_mul( linsolve_yy[j], bmix );   // x_{k+1} = p + bmix * (x_k - x_{k-1})  <-- WAIT, code analysis in session:
                                     // linsolve_yy stores p_{k} - p_{k-1} ?? 
                                     // Re-reading code:
                                     // p.add_mul( linsolve_yy[j], bmix ); // p_{new} = p_Jacobi + bmix * d_old
                                     // linsolve_yy[j] = p_{new} - psa[j]; // d_new = p_{new} - p_old
                                     // psa[j] = p_{new};
```
So: $x_{k+1} = \text{Jacobi}(x_k) + \beta (x_k - x_{k-1})$.
This accelerates the trajectory towards the solution.

**Proposed Implementation for RRsp3**:
We need to introduce persistent buffers `cl_pos_prev_iter` and `cl_quat_prev_iter` (or `cl_solver_momentum_pos/rot`) to store the previous iteration's state or the step difference.
Given memory constraints, storing the *step difference* (`d_pos`, `d_quat`) is preferred (as in `linsolve_yy`).

**Algorithm**:
1.  **Initialize**: `d_pos = 0`, `d_quat = 0`.
2.  **Loop** (Python `step_cluster` or internal):
    *   `pos_pred`, `quat_pred` = JacobiStep(`pos`, `quat`)  (This is `apply_corrections` output)
    *   `pos_new` = `pos_pred` + `beta` * `d_pos`
    *   `quat_new` = `quat_slerp(quat_pred, quat_pred * d_quat, beta)` ?? No, simpler linear approximation for small steps or:
        *   `d_pos_new` = `pos_new` - `pos`
        *   `d_quat_new` = `quat_diff(pos_new, pos)`
        *   `pos` = `pos_new`
        *   `d_pos` = `d_pos_new`

**Rotation Momentum**:
Linear approximation of rotation momentum for quaternions:
$q_{new} = q_{pred} + \beta (q_{curr} - q_{prev})$.
Then normalize. This is cheap and effective for small iterative steps.

## 3. Implementation Plan

### 3.1 Python (`RRsp3.py`)
1.  **Add Buffers**:
    *   `cl_pos_prev`: Stores $x_{k-1}$ (or solver momentum).
    *   `cl_quat_prev`: Stores $q_{k-1}$.
2.  **Update `step_cluster`**:
    *   Allow passing `momentum_beta` (bmix).
    *   Manage buffer swapping or momentum updates.

### 3.2 OpenCL (`RRsp3.cl`)
1.  **Modify `apply_corrections_rigid_ports`**:
    *   Add arguments: `pos_prev`, `quat_prev`, `beta`.
    *   Compute Jacobi prediction `p_jac`, `q_jac` (current logic).
    *   Apply momentum:
        ```c
        float3 p_k = pos[i].xyz;
        float3 p_km1 = pos_prev[i].xyz;
        float3 p_new = p_jac + beta * (p_k - p_km1);
        pos[i].xyz = p_new;
        pos_prev[i].xyz = p_k; // Store old current as prev for next step
        ```
    *   Same for quaternions (linear add + normalize).

## 4. Specific Issues to Address
*   **Weights**: The use of $\alpha = 1/(K dt^2)$ in `RRsp3.cl` is correct for XPBD and equivalent to $M/dt^2$ in PD. No change needed.
*   **Rotation**: `RRsp3.cl` uses small-angle approximation (axis-angle from torque). For solver momentum, we can use linear quaternion interpolation (NLERP) since changes per iteration are small.
